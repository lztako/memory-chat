# Origo AI Assistant — Backend Overview

**Production URL:** https://memory-chat-ochre.vercel.app
**Last updated:** 2026-03-13

---

## What Is This?

Origo AI is a private, invite-only AI assistant built specifically for import/export businesses. It's not a generic chatbot — it learns each company's data, workflows, and preferences over time, making it increasingly difficult to replace (the core lock-in strategy).

Each account represents one company. Origo's team sets up and manages every account manually. Customers never self-register.

## High-Level Architecture

```
Browser (Next.js frontend)
        │
        ▼
POST /api/chat  ──► Build system prompt (memory + skills + docs + files + tasks)
        │
        ▼
Anthropic Claude API (streaming SSE)
        │
        ├── text delta → stream to browser
        └── tool_use  → execute tool → stream result back to Claude
                              │
                              ▼
                    Supabase DB / Tendata API / File cache
```

The entire chat is a **streaming tool loop** — Claude can call multiple tools in sequence before giving a final answer. The browser receives real-time tokens via Server-Sent Events (SSE).

## Database Schema

### Core Tables

**User**
```
id, email, name, createdAt
```
One row per company account. Created by Origo's team, never self-registered.

---

**Conversation + Message**
```
Conversation: id, userId, title, createdAt, updatedAt
Message:      id, conversationId, role (user|assistant), content, createdAt
```
Standard chat history. Title is auto-generated by Claude after the first message.

---

**Memory**
```
id, userId, type, content, importance (1-5), layer (long_term|daily_log),
embedding vector(512), lastUsedAt, createdAt, updatedAt
```
The core personalization engine.
- `long_term` — permanent facts (company name, products, preferences, key contacts)
- `daily_log` — temporary context, reset each day
- `importance 5` is reserved for AI Config entries (user behavior rules)
- Embeddings enable semantic retrieval — the most *relevant* memories to the current message are injected, not just the most recent
- Deduplication on write: if a new memory has cosine similarity < 0.15 to an existing one, it updates instead of creating a duplicate

---

**UserFile**
```
id, userId, fileName, fileType, description, mimeType, size,
rowCount, columns TEXT[], data JSONB, createdAt, updatedAt
```
CSV and XLSX files uploaded by Origo's team on behalf of each customer.
- `data` stores all rows as a JSONB array — flexible schema, no fixed columns
- `columns` is a TEXT array of column names
- `fileType` is a freeform tag (e.g. `shipment`, `invoice`, `contract`, `finance`)
- Queried at runtime via `query_user_file` tool with optional filter/groupBy/aggregate

---

**Task**
```
id, userId, title, description, status (pending|in_progress|done|cancelled),
priority (low|normal|high|urgent), dueDate, linkedCompany, createdAt, updatedAt
```
Action items Claude can create, update, and remind the user about. Overdue and upcoming tasks are automatically injected into every system prompt.

---

**UserSkill**
```
id, userId, name, trigger, solution, tools TEXT[], usageCount,
embedding vector(512), createdAt, updatedAt
```
Auto-generated workflow shortcuts. When Claude solves an unusual problem (strange column name, special data format, custom logic), it saves the solution as a skill so it won't need to re-learn next time. Semantically retrieved based on the current message.

---

**UserAgent**
```
id, userId (nullable), name, description, systemPrompt TEXT,
tools TEXT[], model, maxTurns, isActive, createdAt, updatedAt
```
Sub-agents that Claude can delegate tasks to. `userId = null` means it's a global agent available to all users. Current global agents:
- **Trade Data Analyst** — queries Tendata, summarizes trade flows
- **File Processor** — cleans and analyzes uploaded data files
- **Task Manager** — creates and organizes tasks from conversation context

---

**UserDoc**
```
id, userId, parentId (nullable), parentType (skill|agent|resource),
docType (reference|example|workflow|overview|contact|product),
title, content TEXT, embedding vector(512), createdAt, updatedAt
```
Per-user documents attached to skills, agents, or standalone as "resources". Resources are company-specific knowledge (e.g. shipment workflow, supplier contacts, product specs). Only the title index is injected into every prompt — full content is fetched on-demand via the `read_resource` tool.

---

**GlobalDoc**
```
id, category (company|knowledge), docType, title, content TEXT,
sortOrder, createdAt, updatedAt
```
Origo-wide knowledge base shared across all users. Currently contains Origo's company overview, service offerings, sales playbook, and core philosophy. Injected as index only; Claude fetches full content on demand via `read_global_doc` tool.

---

**GlobalInfo**
```
id, key (unique), value TEXT, sortOrder, createdAt, updatedAt
```
Short, always-injected facts about Origo (company name, industry, country). Kept minimal (4 entries) to avoid token waste on every request.

---

**ConversationContext**
```
id, conversationId (unique), currentTask, quizState JSONB,
pendingItems TEXT[], updatedAt
```
Ephemeral state within a conversation — tracks what the AI is currently doing (e.g. mid-quiz, waiting for confirmation).

---

**UserDashboard**
```
id, userId (unique), widgets JSONB, updatedAt
```
Dashboard widget configuration per user. Widgets are defined as a JSON array and rendered by the frontend using Recharts. Widget types: `kpi`, `progress_kpi`, `bar_chart`, `horizontal_bar`, `line_chart`, `donut_chart`, `table`.

---

**UserTradeData**
```
id, userId, skuTag, dataType (company_list|company_ranking|shipment_records),
tradeDirection (import|export), country, content JSONB, sourceParams JSONB, fetchedAt
```
Cached Tendata query results per user per SKU. Refreshed by Origo's team via the admin trade-sync endpoint.

---

**TendataUsage + TokenUsage**
```
userId, date (YYYY-MM-DD), points/inputTokens/outputTokens
```
Daily usage tracking. Tendata has a hard limit of 500 points/user/day. Claude token usage is tracked but not hard-limited.

## How a Chat Request Works (Step by Step)

1. **Auth** — Supabase JWT verified. User upserted into DB if first visit.

2. **Parallel data fetch** — Before building the prompt, 9 queries run in parallel:
   - Semantic memory retrieval (long_term + daily_log relevant to this message)
   - Conversation history
   - Reminder tasks (overdue + due within 24h)
   - Semantic skill retrieval
   - User file summaries (metadata only, no content)
   - Active tasks list
   - Global Info
   - User resource doc index (titles only)
   - Global doc index (titles only)

3. **System prompt construction** — All data assembled into a structured prompt (~1,500–3,000 tokens typical). Sections: Origo identity → global knowledge index → long-term memory → daily log → AI config → skills → reminders → active tasks → resources → user files → attached files.

4. **History truncation** — DB messages sent to Claude API. Last 4 messages are sent in full; older messages are compressed to 800 characters to prevent context overflow in long conversations.

5. **Claude API streaming** — `anthropic.messages.stream()` with tool_choice: auto. Max 10 tool iterations per request.

6. **Tool loop** — If Claude calls a tool:
   - Tool result is returned and appended to the message list
   - Tool results over 8,000 characters are truncated (except `query_user_file` which is exempt)
   - Claude continues until it produces a final text response or hits the 10-iteration limit

7. **SSE events streamed to browser:**
   - `text` — character-by-character text delta
   - `tool_start` / `tool_done` — tool badge UI
   - `artifact` — chart/table rendered in side panel
   - `folder_write` / `folder_move` — local file system operations
   - `title_update` — conversation title (first message only)

8. **Post-response (async, non-blocking):**
   - Assistant message saved to DB
   - Memory extraction runs via Claude Haiku — identifies new facts to save/update
   - Token usage recorded

## Memory & Personalization System

This is the core differentiator. Every conversation makes Claude smarter about that specific company.

**Extraction:** After each conversation, Claude Haiku reads the full exchange and extracts facts worth remembering (e.g. "company exports white sugar to Europe", "CFO's name is Sarah", "prefers bullet point summaries"). These are stored with importance scores and embeddings.

**Injection:** On the next conversation, the top semantically relevant memories are injected into the system prompt. Claude doesn't say "according to my memory..." — it just knows.

**Deduplication:** New memories are compared against existing ones using vector cosine similarity. If too similar (distance < 0.15), the existing entry is updated rather than creating a duplicate.

**AI Config:** Users (via Origo's team) can set explicit behavior rules — e.g. "always respond in Thai", "assume all prices are FOB unless stated", "user prefers tables over prose". Stored as `type:"user_config"` memories with `importance: 5` (max).

## Tendata Integration

Tendata is a private trade intelligence database. It provides real import/export shipment records — who bought what, from whom, in what quantity, at what price.

- Used **backend-only** — customers never query Tendata directly
- Origo queries it on behalf of each customer for their specific SKUs
- Results are cached in `UserTradeData` table per user per SKU
- Rate limit: 500 points/user/day (enforced server-side)
- 3 tools: `list_trade_companies`, `rank_trade_companies`, `query_trade_data`

## Admin Panel

Accessible at `/admin` (password-gated in browser, Bearer token on API).

Per-user management tabs:
- **Graph** — visual SVG map of the user's data (files, skills, agents, memory, docs)
- **Files** — upload, replace, download, delete CSV/XLSX data files
- **Memories** — view all memory entries
- **Skills** — view auto-generated workflow skills
- **Tasks** — view all tasks
- **Agents** — toggle which global agents are active for this user, add custom agents
- **Resources** — add/view/delete company-specific knowledge docs
- **Widget Config** — configure dashboard JSON

## Security Notes

- All user-facing routes require valid Supabase JWT
- All admin routes require `Authorization: Bearer ADMIN_SECRET` header
- Users can only access their own data (userId scoped on every query)
- No self-registration — accounts created by Origo team only
- Tendata API key is server-side only, never exposed to browser
- File data stored as JSONB in PostgreSQL, not in object storage
